# Copilot Instructions for Next.js Development

このリポジトリでは、保守性・可読性・安全性・レビュー容易性を最優先とする。
実装速度よりも、あとで壊れにくい設計を優先する。

## 基本方針

- Next.js / TypeScript 前提で実装する
- まず「シンプルで読みやすい実装」を選ぶ
- 暗黙的な挙動より明示的な実装を優先する
- 一時的に動くだけの雑な実装を避ける
- 技術的負債を増やす近道は提案しない
- 要件にないことを勝手に実装しすぎない
- 既存コードの流儀に合わせつつ、品質改善の余地があれば安全に提案する

## 出力ルール

- 変更前に、何をどう直すかを短く説明する
- 変更対象ファイルを明示する
- 変更理由を説明できる実装のみ行う
- 大きな変更は一度にやらず、小さく分ける
- リファクタリング時は挙動変更を極力避ける
- 推測で補完せず、不明点はコード上の事実を優先する

## アーキテクチャ方針

- コンポーネントは責務ごとに分割する
- 1ファイルに表示・状態管理・API呼び出し・バリデーションを詰め込みすぎない
- 再利用しない抽象化は作らない
- props の受け渡しが深くなりすぎる場合は設計を見直す
- グローバル状態は本当に必要な場合のみ導入する
- UI とデータ取得ロジックを可能な限り分離する
- 画面実装とビジネスロジックを分離する
- サーバー側で行うべき処理とクライアント側で行う処理を混在させない

## ディレクトリ設計の考え方

- ルーティング関連の責務と UI コンポーネントの責務を分離する
- UI部品は `components/` に分ける
- ドメインロジックやユーティリティは `lib/` や `features/` に分ける
- API通信処理はUIから分離する
- 型定義は使う場所の近くに置き、広く使うものだけ共通化する
- server-only な処理と client-only な処理を混在させない

## TypeScript ルール

- `any` は原則禁止
- 型アサーション (`as`) は最小限にする
- 型推論で十分な箇所は冗長に書きすぎない
- ただし公開API、props、戻り値は明示したほうが読みやすい場合は明示する
- union type を乱暴に扱わず、分岐で安全に絞り込む
- nullable な値は必ず考慮する
- 「とりあえず optional」にしない
- 型で防げる不具合は型で防ぐ

## React / Next.js ルール

- 不要な `useEffect` を使わない
- `useEffect` は外部同期、副作用、購読、DOM操作など本当に必要な用途に限定する
- props から導出できる値を state に入れない
- state は最小限に保つ
- イベントハンドラは意味のある名前を付ける
- JSX 内に複雑なロジックを書きすぎない
- 繰り返し使う分岐や整形は事前に変数や関数へ切り出す
- key は index ではなく安定した識別子を使う
- `useMemo` / `useCallback` は必要性が明確な場合のみ使う
- パフォーマンス最適化を儀式的に入れない
- Next.js のデータ取得方式は採用している構成に合わせる
- Pages Router / App Router の違いを勝手に持ち込まない

## データ取得・API ルール

- fetch 処理は責務を分離する
- 通信失敗時・空データ時・ローディング時を必ず考慮する
- APIレスポンスを信用せず、必要に応じてバリデーションする
- フロントエンドで環境変数や秘密情報を露出しない
- クライアントから直接叩く必要がない処理はサーバー側に寄せる
- 例外処理は握りつぶさず、ユーザー向け表示と開発者向け原因を分けて考える

## フォーム実装ルール

- 入力値検証は明示的に行う
- バリデーションルールは UI に散らさず整理する
- エラーメッセージはユーザーに理解できる文言にする
- submit 中の二重送信を防ぐ
- 成功時・失敗時の状態遷移を明確にする
- フォームが大きい場合は責務を分割する

## UI 実装ルール

- 見た目のためだけのハードコードを増やしすぎない
- マジックナンバーを避け、意味のある単位や定数化を検討する
- className が長くなりすぎる場合は分割を検討する
- アクセシビリティを意識する
- button には適切な `type` を付ける
- form 要素には label を関連付ける
- クリック可能要素に div を多用しない
- 画像には適切な alt を付ける
- ローディング中・空状態・エラー状態のUIを省略しない

## 命名規則

- 変数名・関数名は役割が分かる名前にする
- `data`, `item`, `temp`, `handleSomething` のような曖昧すぎる命名を避ける
- 真偽値は `is`, `has`, `can`, `should` で始める
- 関数名は動詞から始める
- 略語を多用しない
- 単数形・複数形を正しく使い分ける

## 条件分岐・可読性ルール

- ネストを深くしすぎない
- 早期 return を使って見通しを良くする
- 三項演算子の入れ子を避ける
- 長い if 文は意味ごとに分割する
- 読みにくいワンライナーを書くくらいなら複数行に分ける

## エラーハンドリング

- 失敗しうる処理は必ず失敗時を考慮する
- `console.error` だけで済ませない
- ユーザーに見せるエラーと内部調査用エラーを分ける
- 予期可能な失敗は UI でハンドリングする
- 例外メッセージをそのままユーザーに露出しない

## セキュリティ

- シークレットはサーバー側でのみ扱う
- `NEXT_PUBLIC_` を付ける値は公開される前提で扱う
- ユーザー入力を信用しない
- URLパラメータ、フォーム値、外部APIレスポンスは検証する
- `dangerouslySetInnerHTML` は原則避ける
- 使う場合は安全性を確認し、理由を明示する

## パフォーマンス

- まず不要な再レンダリングを生む設計を避ける
- 重い処理は適切な場所へ分離する
- リスト描画は安定した key を使う
- 不必要な状態更新を避ける
- 最適化は計測または根拠がある場合に行う

## テストしやすさ

- 副作用の強い処理は分離する
- 純粋関数に切り出せるロジックは切り出す
- UI とロジックを分離してテストしやすくする
- モック前提でしか動かない設計を避ける
- 重要な分岐や整形処理にはテスト追加を検討する

## レビューで指摘されやすい実装を避ける

- 不要な `useEffect`
- props から導出可能な state
- `any` の使用
- null 考慮漏れ
- index を key に使う
- 1コンポーネントに責務を詰め込みすぎる
- API 呼び出しを直接 UI にベタ書き
- エラー処理・ローディング処理不足
- 命名が曖昧
- 長すぎる関数
- 長すぎる JSX
- 一貫性のないファイル構成
- server/client の責務混在
- その場しのぎの型アサーション
- コメントがないと意図が伝わらない複雑実装

## 実装時の優先順位

以下の順で判断すること。

1. 正しさ
2. 可読性
3. 保守性
4. セキュリティ
5. パフォーマンス
6. 開発速度

## Copilot への具体的な振る舞い要求

- コードを提案する際は、まず既存構成に合わせる
- ただし既存構成が明らかに悪い場合は、より良い分割案も併記する
- 複雑な実装を提案する場合は、より単純な代替案がないか先に検討する
- 変更はレビューしやすい単位で提案する
- 可能なら「最小変更案」と「改善案」の2段階で提案する
- ライブラリ追加は安易に行わない
- 一貫性のない命名や設計を増やさない
- 既存の lint / format / test 方針がある場合は従う
